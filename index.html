Git笔记

1.git是分布式版本控制工具
2.git分为工作区->暂存区->本地仓库
3.git基本操作

  • 1.初始化本地仓库 git init .git文件
  • 2.设置签名
    用户级别
    git config user.name XXX
    git config user.email xxx
    系统级别
    git config -global user.name xxx
    git config -global user.email xxx
  • 3.加入暂存区 git add [fileName]
  • 4.提交到本地库 git commit -m “xxx” [fileName]
  • 5.查看仓库状态 git status
  • 6.回退上一级状态 git rm –cached
  • 7.查看历史版本 git log /git relog git /git log –oneline(只显示过去)

  • 8.版本回退
    索引操作
    git reset –hard index
    使用^符号后退
    git reset –hard HEAD^
    使用~符号后退
    git reset –hard HEAD~n

  • 9.比较文件 git diff fileName;git diff HEAD;git diff HEAD fileName

4.HEAD指针移动方式

  • 1.soft本地库移动
  • 2.mixed本地库移动，暂存区移动
  • 3.hard本地库移动，暂存区移动，工作区移动

5.删除文件并找回

  • 1.git init
  • 2.vim fileName
  • 3.git add fileName
  • 4.git commit fileName
  • 5.vim fileName
  • 6.rm fileName
  • 7.git reset –hard index
    or
  • 1.git init
  • 2.vim fileName
  • 3.git add fileName
  • 4.git commit fileName
  • 5.rm fileName
  • 6.git reset –hard HEAD

6.分支操作

  • 1.查看分支 git branch -v
  • 2.创建分支 git add branch branchName
  • 3.切换分支 git checkout branchName
  • 4.合并分支 git merge branchName->B A 合并 B
  • 5.master branch：主分支
  • 6.develop branch：开发分支
  • 7.feature branch：功能分支
  • 8.release branch：预发布分支
  • 9.hotfix branch：bug修复分支

7.分支冲突

  • 1.原因:A分支修改文件提交 B分支修改同一个文件提交 仓库不知道以哪个为标准
  • 2.解决办法:
    编辑文件删除特殊符号
    git add fileName
    git commit -m “” 不带文件名

8.Git原理

  • 1.Hash 算法:SHA-1算法
  • 2.版本控制:链表
  • 3.分支管理:HEAD指针指向分支

9.协同开发

  • 1.本地库上传到远程库
    git remote -v 查看远程库
    git remote add XXX https://wwww.xxxx.git 命名远程库
    git push xxx master(branch) 上传到远程库
  • 2.远程库克隆到本地库
    git clone https://www.xxxx.git
    效果:完整将远程库下载到本地库且初始化了本地库
  • 3.加入成员从本地库上传到远程库
    1.setting->Collaborators->email
    2.复制邀请链接发送给邀请人->accept
    3.git push xxx xxx
  • 4.从本地库拉取项目
    git pull xxx xxx / git fetch xxx xxx pull=fetch+merg

10.跨团队开发

  • 1.fork
  • 2.pull->本地修改推送到远程
  • 3.pull request->new pull request

    插件->目标（命令）->生命周期

10.依赖:解析依赖信息时会到本地仓库中查找被依赖的Jar包,对于我们开发的Maven工程，使用install插件可以进入本地仓库查找，若不存在则会报错。

先install依赖的模块
11.范围依赖

  • 1.complie 对主程序有效，测试程序有效，参与打包部署 例如:spring
  • 2.test 对主程序无效，测试程序有效，不参与打包部署 例如:junit
  • 3.provide 对主程序有效，测试程序有效，不参与打包部署，servlet容器中有。例如:servlet-api jstl-api jsp-api

12.依赖传递性

  • 1.直接依赖 例如:上下级
  • 2.间接依赖 例如:上级下下级
  • 3.非compile范围不能依赖传递 继承出现的原因

13.依赖排除

14.模块Jar包冲突

  • 1.最短路径优先原则
  • 2.相同路劲pom.xml中优先声明原则
  • 3.解决办法:版本统一

15.继承

  • 1.原因：由于test范围不能传递所以造成在各个模块中出现版本不一致
  • 2.作用:统一管理各个模块的test依赖
  • 3.方法：
  • 1.父工程统一test依赖版本
  • 2.子工程删除test依赖版本
  • 3.子工程声明父工程引用
  • 4.子工程删除和父工程相同的坐标
  • 1.清理->编译->测试->报告->安装（若不通过则不可以打包）->打包->部署
  • 2.注意:依赖父工程的子工程要先install父工程再install子工程 聚合出现的原因

16.版本统一

17.聚合:一键安装各个模块(从属父子依赖工程不用先安装父工程再安装子工程，可以一起安装不分顺序)

18.优点:

  • 1.将项目划分为各个模块高内聚低耦合
  • 2.不要到官网复制粘贴jar包通过坐标gav将jar包从远程库下载到本地库而项目只需要一个pom.xml文件即可不用浪费空间

19.缺点

  • 1.若项目过于庞大,需要的坐标过多则pom.xml文件就会变得臃肿

20.选择

  • 1.小项目用Maven自动化构建工程。
  • 2.大项目用Gradle自动化构建工程。

选择排序算法

选择排序算法稳定性:不稳定。
选择排序算法时间复杂度:平均时间O(n²),最坏情况O(n²),最好情况O(n)。
选择排序算法空间复杂度:O(l)。
C++代码手写实现从小到大排序选择排序算法

总结:
1.思想:根据找最小值来确定是否进行交换。
2.注意:在外循环中循环次数要-1,因为最后一个数不用比较;在内循环中当找到比min更小的数下标值将此下标赋值给min。
3.选择排序算法是不稳定算法;时间复杂度平均时间和最坏情况都是O(n²);最好情况是O(n);空间复杂度O(l),以时间换空间。

冒泡排序算法

冒泡排序算法稳定性:稳定。
冒泡排序算法时间复杂度:平均时间O(n²),最坏情况O(n²),最好情况O(n)。
冒泡算法空间复杂度:O(l)。
C++代码手写实现从小到大冒泡排序算法

总结
1.思想:冒泡算法就是两两比较将最大(最小)的值沉于底下,外循环次数是循环元素的个数次,内循环次数随着最大(最小)的值沉于底下而减少,每次内循环次数随i的增大而减少。
2.注意:内循环需要n-1因为两两比较每次比较的个数都会比元素个数-1。
3.冒泡排序是一种用时间换空间的排序方法,n越小时越好。
4.最坏情况是把顺序的排列变成逆序,或者把逆序的数列变成顺序，最差时间复杂度O(n²)只是表示其操作次数的数量级。
5.最好的情况是数据本来就有序,复杂度为O(n)。

快速排序算法

快速排序算法稳定性：不稳定。
快速排序算法时间复杂度:平均时间O(nlog2n),最坏情况O(n²),最好情况O(nlog2n)。
快速排序算法空间复杂度:O(log2n)~O(n)。
C++代码手写实现从小到大快速排序

总结：
1.思想:分治法,选取基准值来进行筛选,分成小于基准值的部分和大于基准值的部分,再将其两部分用同样的方法进行递归。冒泡算法的扩展。
2.注意:时刻清楚i,j值下标的位置,不能让i<j的值。若等于则将基准值赋予i下标的数组值。
3.n大时好,快速排序比较占用内存,内存随n的增大而增大,但却是效率高不稳定的排序算法。
4.划分之后一边是一个,一边是n-1个,这种极端情况的时间复杂度就是O(n²)。
5.最好的情况是每次都能均匀的划分序列,O(n*log2n)。
