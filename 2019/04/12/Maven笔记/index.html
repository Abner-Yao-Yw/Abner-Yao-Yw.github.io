9.生命周期;插件;目标
Maven生命周期

  • 1.clean 清理
  • 2.validate 核实
  • 3.compile 编译
  • 4.test 测试
  • 5.package 打包
  • 6.verify 核实
  • 7.install 安装
  • 8.site
  • 9.deploy 部署
    插件
  • 1.clean
  • 2.install
  • 3.deploy
  • 4.site
    目标(命令)
  • 1.mvn clean 清理
  • 2.mvn compile 编译主程序
  • 3.mvn test 执行测试程序
  • 4.mvn test compile 编译测试程序
  • 5.mvn package 打包
  • 6.mvn deploy 部署
  • 关系:
    插件->目标（命令）->生命周期

10.依赖:解析依赖信息时会到本地仓库中查找被依赖的Jar包,对于我们开发的Maven工程，使用install插件可以进入本地仓库查找，若不存在则会报错。

先install依赖的模块
11.范围依赖

  • 1.complie 对主程序有效，测试程序有效，参与打包部署 例如:spring
  • 2.test 对主程序无效，测试程序有效，不参与打包部署 例如:junit
  • 3.provide 对主程序有效，测试程序有效，不参与打包部署，servlet容器中有。例如:servlet-api jstl-api jsp-api

12.依赖传递性

  • 1.直接依赖 例如:上下级
  • 2.间接依赖 例如:上级下下级
  • 3.非compile范围不能依赖传递 继承出现的原因

13.依赖排除

14.模块Jar包冲突

  • 1.最短路径优先原则
  • 2.相同路劲pom.xml中优先声明原则
  • 3.解决办法:版本统一

15.继承

  • 1.原因：由于test范围不能传递所以造成在各个模块中出现版本不一致
  • 2.作用:统一管理各个模块的test依赖
  • 3.方法：
  • 1.父工程统一test依赖版本
  • 2.子工程删除test依赖版本
  • 3.子工程声明父工程引用
  • 4.子工程删除和父工程相同的坐标
  • 1.清理->编译->测试->报告->安装（若不通过则不可以打包）->打包->部署
  • 2.注意:依赖父工程的子工程要先install父工程再install子工程 聚合出现的原因

16.版本统一

17.聚合:一键安装各个模块(从属父子依赖工程不用先安装父工程再安装子工程，可以一起安装不分顺序)

18.优点:

  • 1.将项目划分为各个模块高内聚低耦合
  • 2.不要到官网复制粘贴jar包通过坐标gav将jar包从远程库下载到本地库而项目只需要一个pom.xml文件即可不用浪费空间

19.缺点

  • 1.若项目过于庞大,需要的坐标过多则pom.xml文件就会变得臃肿

20.选择

  • 1.小项目用Maven自动化构建工程。
  • 2.大项目用Gradle自动化构建工程。
